## Decision (Day 5 / T-3, 30 April 2026)

The task module's service layer ships **bimodal** rather than CRUD-symmetric, AND ships only methods with an MVP caller.

**System-only methods** (run under `withServiceRole`, no `requirePermission`, audit emits with `actorKind: "system"`):
- `createTask(ctx, input) ‚Üí Task`
- `bulkCreateTasks(ctx, inputs) ‚Üí BulkCreateResult` (all-or-nothing)

**User-flow methods** (run under `withTenant` + `requirePermission`):
- `getTask(ctx, id) ‚Üí Task | null`            ‚Äî `task:read`
- `listTasks(ctx) ‚Üí readonly Task[]`           ‚Äî `task:read`
- `updateTask(ctx, id, patch) ‚Üí Task`          ‚Äî `task:update`

**No `deleteTask` method ships.** Five methods, not six. The reasoning extends the same principle that excludes user-facing `task:create`: ship what has an MVP caller, defer everything else.

The existing permissions catalogue at `src/modules/identity/permissions.ts` is **NOT expanded.** No `task:create`, `task:delete`, or `task:bulk_create` user-facing permissions are added. The catalogue's existing comment ("No `task:create` / `:delete` ‚Äî tasks are generated by the nightly batch") still holds.

**Why no user-facing create:** The original Day-2 design (cron generates, users only update / read) was a deliberate, principled call grounded in plan ¬ß4.6.2. Day-5's brief drift toward user-facing CRUD was reviewed and the catalogue's narrower surface won. There is no MVP caller for user-driven task creation:
- The cron creates tasks under `withServiceRole` directly, no user permission needed.
- The migration-import flow creates tasks gated behind the existing `tenant:migration_gate_set` permission (Transcorp Systems Team only).
- Manual task entry from the UI is **not** in pilot scope.

**Why no `deleteTask` (added in T-3 review):** Same principle, applied a level deeper. No MVP caller for task deletion exists:
- The cron does not delete tasks ‚Äî failed pushes go through the failed_pushes DLQ table (T-6/T-7), not deletion + retry.
- The migration-import flow is forward-only; rollback is sysadmin SQL, not a service method.
- Administrative cleanup of misrouted imports happens out-of-band via Supabase SQL when rare.
- No GDPR / right-to-erasure flow in pilot.

Adding a service method for an absent caller would be speculative scope. If a real requirement surfaces post-pilot (GDPR erasure becomes binding, manual cleanup of misrouted imports becomes routine, a partner-API requires task deletion), `deleteTask` lands then with explicit scope and an audit event added in the same PR. The audit event `task.deleted` is **not** added to the vocabulary today; it lands when the method does.

**T-5 scope correction:** the brief's "6 endpoints (single + bulk)" reflected the same drift. T-5 ships only THREE routes:
- `GET /api/tasks` (requires `task:read`)
- `GET /api/tasks/:id` (requires `task:read`)
- `PATCH /api/tasks/:id` (requires `task:update`)

No user-facing `POST /api/tasks` or `DELETE /api/tasks/:id`. The bulk-create surface lives behind the migration-gate flow when that wires up (Day 6+).

## Audit vocabulary additions (Day 5 / T-3)

One new event type added to `src/modules/audit/event-types.ts`, `systemOnly: true`:
- `task.bulk_created` ‚Äî meta-event emitted by `bulkCreateTasks` alongside per-task `task.created` events. Carries `task_ids[]` and `count`.

`task.deleted` was drafted in the initial T-3 commit but **removed in review** alongside `deleteTask`. The two ship together when (and if) the method does.

The vocabulary addition is scoped to where it's used (T-3 service), **not** a permission-catalogue expansion.

## Future-expansion path (post-pilot)

If a real user-create requirement surfaces post-pilot (e.g., direct manual task entry in a tenant UI, or a partner-API task creation endpoint distinct from the cron / migration-import paths):

1. Add `task:create` to the permissions catalogue with appropriate role assignments (Tenant Admin? Operator? Driver? Decided then, not now).
2. Flip `task.created`'s `systemOnly` to `false` OR add a separate `task.created_manually` event to distinguish system-actor and user-actor flows in audit reporting.
3. Update the catalogue's `// ---- task ----` comment to reflect the broader surface.
4. Add a service-layer branch in `createTask` so `requirePermission("task:create")` runs for user actors and `assertSystemActor` only fires for system actors that lack the permission.

If a real `deleteTask` requirement surfaces (GDPR erasure, manual cleanup, partner-API):

1. Add `task:delete` to the permissions catalogue (or keep system-only, depending on caller).
2. Add `task.deleted` to the audit event-types vocabulary (`systemOnly` flag matches the permission decision).
3. Add `deleteTask(ctx, id)` to the service with the same shape as the C-3 consignees `deleteConsignee` (pre-fetch, audit metadata captured pre-delete, post-commit emit).

The expansion path is well-defined; this decision freezes the pilot's surface, not the post-pilot one.

## Why this matters for review

Reviewers reading T-3's PR may be surprised that the brief's `requirePermission("task:create")` line is not implemented as written. This memory captures the design call so the deviation is principled and traceable, not silent drift in the other direction.
